Je demande à pi de me dire comment il gère le degré de "thinking" qu'un modèle déploie. Voici sa réponse.

Pi controls the thinking/reasoning budget through token limits that are sent to the LLM provider's API. Here's the mechanism:

1. Configuration Levels
Pi supports these thinking levels: off, minimal, low, medium, high

2. Token Budget Mapping
The thinking level translates to specific token budgets. The default mapping (configurable via thinkingBudgets in settings):

{
  "thinkingBudgets": {
    "minimal": 1024,
    "low": 4096,
    "medium": 10240,
    "high": 32768
  }
}

3. How It's Applied
Pi sends the thinking budget to the LLM provider via API parameters. For models that support extended thinking (like Claude, Gemini, etc.), this is typically sent as:

- max_tokens for thinking/reasoning (separate from output tokens)
- Provider-specific parameters like thinking.budget (Anthropic) or thoughts (Google)

The thinking budget directly controls how many tokens the model can use for internal reasoning before producing its actual response - higher budgets allow for more complex problem-solving but use more tokens (cost) and take longer.

Kyutai sort Hibiki-Zero, un nouveau modèle de traduction vocale multilingue en temps réel. Hibiki-Zero traduit à partir de 4 langues (dont le français) vers l'anglais. Kyutai annonce "précision, faible latence, qualité audio élevée". En outre, la voix traduite est un clone de la voix source. Ils utilisent une méthode d'entraînement basée sur l'apprentissage par renforcement et toutes sortes d'astuces pour réduire la latence et améliorer la fluidité.

Le système s'appuie sur le codec audio Mimi, qui transforme les ondes sonores en séquences de tokens sémantiques et acoustiques pouvant être traités comme du texte par le transformer. Il s'agit de traduction "zéro-shot" : lors de l'entraînement du modèle, le besoin de données alignées mot à mot n'est plus nécessaire. Le modèle apprend par lui-même la meilleure stratégie de traduction et fonctionne avec un système de récompenses (cf. DeepSeek-R1-Zero).

Pour tester Hibiki-Zero, rien de plus simple : avec uv installé, il suffit d'utiliser la commande suivante uvx -p 3.13 hibiki-zero serve --gradio-tunnel. On obtient un lien vers une application web exécutable en local. Ne disposant pas d'un GPU je fais l'essai sur Google Colab, qui propose maintenant un terminal. Le résultat est moyen, ça fonctionne un peu au début, mais avec une grosse latence et beaucoup d'hallucinations (le modèle se met à parler tout seul et raconte n'importe quoi). Les config proposées par Colab ne sont pas assez puissantes.

Je me dis que ça vaudrait le coup de tester Hibiki-Zero sur une machine appropriée et me renseigne auprès de ChatGPT et Claude sur la démarche à suivre. Après un temps infini passé à questionner les IA, j'arrive à la conclusion qu'il me faut une machine équippée d'un GPU de type RTX 3090 ou 4090 avec 24 GB VRAM. Côté fournisseur, RunPod semble être un bon choix.

Je questionne Claude sans fin sur les différents aspects de la démarche, autant techniques que financiers. Il y a en effet quelques "pièges" potentiels quand on utilise ces services, le plus saillant étant la gestion de l'espace disque. Il faut bien comprendre la distinction entre "Container Disk", "Volume Disk" et "Network Volume", et s'assurer de ne rien laisser en mode persistant une fois la session terminée, sous peine d'avoir une mauvaise surprise à la fin du mois.

Les sessions avec pi s'enchaînent et me permettent d'améliorer mes applis basées sur la voix. Il y en a maintenant trois : voice2prompt, voice2llm_local et voice2llm_groq. Les deux dernières sont inspirées comme dit précédemment des travaux de Kyutai en matière de speech-to-text (STT) et de text-to-speech (TTS) combinés avec un LLM au milieu.

Comme la partie STT ne fonctionne pas sur mon ordi, je la remplace par Whisper, avec le code de voice2prompt. Cette version semi-locale fonctionne plutôt mieux, avec moins de latence que la version 100 % locale.

L'écueil principal est la latence entre la sortie du LLM et sa reconstitution sonore. Plusieurs stratégies sont utilisées pour tenter de résoudre ce problème, avec notamment l'idée de "streamer" les phrases ou les mots par "chunks" dans l'entrée du modèle TTS plutot que lui balancer le texte en un seul bloc. Mais cela ne fonctionne que partiellement.

En revanche, je fais une grande découverte : je trouve enfin un moyen de ne plus utiliser la version standard de PyTorch (ce que fait Pocket-TTS par ex.) qui est énorme et télécharge toutes sortes de bibliothèques CUDA compatibles avec les machines équipées de GPU. Si comme moi vous avez une modeste machine sans GPU, il est préférable d'utiliser pytorch-cpu, bibliothèque ultra-légère qui est téléchargée par uv en un clin d'oeil.

Pour cela, il faut modifier les "inline scripts" de uv comme suit.

Avant :

# /// script
# requires-python = ">=3.10"
# dependencies = [
#     "numpy",
#     "torch",
# ]
# ///

Après :

# /// script
# requires-python = ">=3.10"
# dependencies = [
#     "numpy",
#     "torch",
# ]
#
# [tool.uv]
# extra-index-url = ["https://download.pytorch.org/whl/cpu"]
# ///

"Over the past few months, Claude Code has turned into a spaceship with 80% of functionality I have no use for. The system prompt and tools also change on every release, which breaks my workflows and changes model behavior. I hate that."

Il explique que le prompt "système" de pi est très court. Tout est fait pour ne pas surcharger inutilement le contexte du modèle.

"It turns out that all the frontier models have been RL-trained up the wazoo, so they inherently understand what a coding agent is. There does not appear to be a need for 10,000 tokens of system prompt. pi's system prompt and tool definitions together come in below 1000 tokens."

Concernant la sécurité et l'utilisation du mode "YOLO", il explique que la plupart des soit-disant mesures de sécurité des autres outils populaires sont de la poudre au yeux ("security theater").

"pi runs in full YOLO mode and assumes you know what you're doing. It has unrestricted access to your filesystem and can execute any command without permission checks or safety rails. No permission prompts for file operations or commands. No pre-checking of bash commands by Haiku for malicious content. Full filesystem access. Can execute any command with your user privileges.

If you look at the security measures in other coding agents, they're mostly security theater. As soon as your agent can write code and run code, it's pretty much game over."

Concernant d'autres fonctionalités comme les MCP, les processus lancés en arrière plan ("background bash") ou les "sub-agents", Zechner rejette tout en bloc par souci de simplicité et d'efficacité.

"Popular MCP servers like Playwright MCP (21 tools, 13.7k tokens) or Chrome DevTools MCP (26 tools, 18k tokens) dump their entire tool descriptions into your context on every session. That's 7-9% of your context window gone before you even start working. Many of these tools you'll never use in a given session."

Il faut ensuite copier le code ci-dessous et le coller dans le fichier de configuration du client utilisé. J'ai fait l'essai avec Gemini CLI et ai donc ajouté ces instructions au fichier settings.json de Gemini.

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": [
        "@playwright/mcp@latest"
      ]
    }
  }
}

Problème : le fichier settings.json n'est pas vide. Il contient déjà du code et il faut insérer le code ci-dessus dans l'existant sans se tromper au niveau des accolades et des virgules, ce qui demande pas mal d'attention. Au final, dans mon cas, ça ressemble à ça :

{
  "ide": {
    "hasSeenNudge": true
  },
  "security": {
    "auth": {
      "selectedType": "oauth-personal"
    }
  },
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": [
        "@playwright/mcp@latest"
      ]
    }
  }
}

Une fois ces préparatifs terminés, on peut lancer Gemini CLI, qui détecte automatiquement qu'un MCP server a été ajouté. On peut maintenant taper des instructions comme "ouvre le navigateur et rends toi sur le site de telerama.fr", par exemple.
